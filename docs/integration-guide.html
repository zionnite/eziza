Delivery API Integration Guide

Eziza is a delivery marketplace, not a dispatch service — nearby riders and logistics companies bid on your delivery, you (or your buyer) pick an offer, then Eziza handles live GPS tracking, OTP-verified handoff, and payout. This guide covers integrating your own platform with it.

This guide is for an engineer integrating a platform — an e-commerce checkout, for example — with Eziza on behalf of their own buyers and merchants. ZeeFashion isn't a special case here; it's the first tenant built against exactly this API.

Terminology: "bid" = "offer"

The Eziza app shows riders/companies submitting offers on a delivery — that's the user-facing word. The API and data model underneath still say bid everywhere (accept-bid, bid_id, bid.placed). They're the same concept; the API just predates the app's own copy change. Read "bid" as "offer" throughout this guide.

01Getting access

Eziza doesn't have self-service tenant signup yet. Email admin@eziza.online with your platform name and a contact email, and we'll create your tenant and issue your first API key from the internal admin console. You'll receive:

  • A tenant record — name, contact email, and an optional webhook URL for receiving events
  • An API key — shown to us once at creation, then never retrievable again. Store it like any other secret.

You can request additional keys later (e.g. one for staging, one for production), or have one revoked at any time.

No sandbox environment

Test keys hit the same project as production today. Use a dedicated key per environment, and prefix your external_order_id so test deliveries are identifiable.

02Authentication

Every request — except incoming webhooks, which are signed instead of bearer-authenticated (see Section 5) — needs your API key as a bearer token:

Authorization: Bearer <your_api_key>

A missing, wrong, revoked key, or a key belonging to a deactivated tenant, gets a 401:

{ "error": "Unauthorized" }

03Delivery lifecycle

A delivery moves through one of two paths:

open→ assigned→ awaiting_pickup_confirm→ picked_up→ delivered→ confirmed
openor assigned→ cancelled
StatusMeaningWho moves it here
openLive on the job board, accepting bidscreate-delivery
assignedA bid was accepted, a rider/company is assignedaccept-bid
awaiting_pickup_confirmRider has arrived, waiting on your merchant to hand the package overthe rider's own app
picked_upHandoff confirmed, rider is en routeconfirm-pickup
deliveredRider marked it delivered, awaiting OTP-verified confirmationthe rider's own app
confirmedReceipt confirmed — the rider/company gets paid at this pointconfirm-receipt
cancelledCancelled while still open or assignedcancel-delivery

open deliveries accept bids for a bid window (30 minutes by default — an Eziza-side setting, not per-request) before closing to new offers. bid_closes_at in the create-delivery response tells you when.

GET/get-delivery?id=<delivery_id>

Full delivery status, assigned rider info, and live GPS if a rider is currently active on it.

Response 200 — every column on the delivery row, plus:

  "external_order_id": "…",
  "status": "picked_up",
  "agreed_price": 2500,
  "rider": {
    "id": "uuid",
    "full_name": "…",
    "phone": "…",
    "vehicle_type": "bike",

    "rating_avg": 4.8
  },
  "rider_location": { "latitude": 6.5244, "longitude": 3.3792, "updated_at": "…" }

rider is null until a bid is accepted. rider_location is null unless the delivery is assigned, awaiting_pickup_confirm, or picked_up and the rider currently has a live GPS row. 404 if the delivery doesn't exist or doesn't belong to your tenant.

{ "delivery_id": "uuid", "reason": "optional string" }

Only works while status is open or assigned — 409 otherwise.

Response 200

{ "delivery_id": "uuid", "status": "cancelled" }

POST/accept-bid

Accepts one bid on an open delivery; every other pending bid on it is automatically rejected.

{ "delivery_id": "uuid", "bid_id": "uuid" }

Response 200

{ "delivery_id": "uuid", "bid_id": "uuid", "status": "assigned", "rider_id": "uuid", "agreed_price": 2500 }

409 if the delivery isn't open anymore; 404 if the bid doesn't exist, isn't on this delivery, or was already accepted/rejected.

Current limitation

There's no endpoint to list bids on a delivery. The only way to discover them today is the bid.placed webhook — if you need a polling fallback, ask us for a list-bids endpoint rather than assuming one exists.

POST/confirm-pickup

Call this when your merchant has physically handed the package to the rider. Only valid while status is awaiting_pickup_confirm.

{ "delivery_id": "uuid" }

Response 200

{ "delivery_id": "uuid", "status": "picked_up" }

POST/confirm-receipt

Call this when your buyer confirms they've received the package. Only valid while status is delivered. This is what triggers the rider/company getting paid.

{ "delivery_id": "uuid" }

Response 200

{ "delivery_id": "uuid", "status": "confirmed" }

05Webhooks

If your tenant record has a webhook_url set, Eziza POSTs an event to it whenever something changes. Every request carries:

Content-Type: application/json
X-Eziza-Event: <event name>
X-Eziza-Signature: <hex HMAC-SHA256 of the raw request body>

Verify the signature before trusting the payload — compute HMAC-SHA256 of the exact raw body bytes using your WEBHOOK_SIGNING_SECRET (given to you alongside your API key), hex-encode it, and compare to X-Eziza-Signature with a constant-time comparison:

// Node.js
const crypto = require('crypto');
function isValid(rawBody, signatureHeader, secret) {
  const expected = crypto.createHmac('sha256', secret).update(rawBody).digest('hex');
  return crypto.timingSafeEqual(Buffer.from(expected), Buffer.from(signatureHeader));
}
No retries

If your endpoint is down or returns a non-2xx, the event is logged internally but not retried. Treat webhooks as a fast-path notification, not a guaranteed delivery mechanism — poll get-delivery periodically as a fallback for anything time-sensitive, exactly like Eziza's own rider app does internally for missed realtime events.

06Errors

StatusMeaning
400Missing required field(s)
401Missing/invalid/revoked API key, or your tenant is deactivated
404Not found, or exists but belongs to a different tenant
409Valid request, but the delivery isn't in a state that allows this action
500Unexpected error — including a reused external_order_id

All error bodies are { "error": "human-readable message" }.
